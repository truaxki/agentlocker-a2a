## Issues Found 🔴

### 1. Chrome Profile Lock on Repeated Launches
**Symptom:** `Failed to create a ProcessSingleton for your profile directory`  
**Root cause:** `launch_persistent_context` locks the profile dir. If a prior Chrome for Testing instance didn't fully exit, the next launch fails.  
**Fix options (in order of preference):**
- **Option A (preferred):** Switch to CDP connect mode. Start Chrome once manually with `--remote-debugging-port=9222`, then use `playwright.chromium.connect_over_cdp("http://localhost:9222")` for all tasks. No profile lock issues. Uses real browser with real cookies.
- **Option B (fallback):** Add a cleanup step at the start of every skill execution: `pkill -f "Google Chrome for Testing"` before launching.
- **Option C (quick fix):** Run `pkill -f "Google Chrome for Testing"` manually when stuck, then ALSO delete the lock file: `rm /Users/kirk/operator-prototype/chrome-profile/SingletonLock`. The process dying does not remove the lock file — both steps are required.
- **Option D (permanent fix):** Add to server startup code: delete `SingletonLock` at `{profile_dir}/SingletonLock` before every `launch_persistent_context` call.

### 2. No Artifact URL Returned on Error
**Symptom:** Error responses don't include a screenshot or partial result  
**Improvement:** Even on failure, capture and return a screenshot of whatever state Chrome was in before crashing. Helps Astra diagnose remotely.
